## 一、项目概述

### 1.1 项目简介

本项目构建了一个**由 7 个专业角色 Agent 组成的多智能体协作系统**，包含规划 Agent、检索 Agent、执行 Agent、编程 Agent、写作 Agent、评审 Agent 和总结 Agent。系统能够自动完成编程开发、报告撰写、数据分析、知识问答等多种复杂任务，强调多 Agent 调度与工具链协同。

用户只需用自然语言描述需求，系统通过 **Router 意图分类器**自动识别任务类型和复杂度，由 **LangGraph 状态图**编排多个 Agent 按流程协作，最终输出结构化任务报告与完整执行日志。

### 1.2 核心能力一览

| 能力维度 | 实现方式 |
|----------|----------|
| **多角色协作** | 7 个专业 Agent，各司其职，由 LangGraph StateGraph 统一编排 |
| **意图识别** | Router LLM 自动分类 5 种任务类型 + 轻/重复杂度 + 关键词覆写安全网 |
| **任务拆解** | Planner Agent 自动将复杂任务拆解为可执行步骤清单 |
| **知识检索** | ChromaDB 向量数据库 + BAAI/bge-small-zh-v1.5 嵌入模型，支持 RAG |
| **代码执行** | subprocess 沙箱隔离执行，exitcode 驱动确定性分支路由 |
| **结果校验** | Tester Agent 代码评审 + 文档质量检查 + ≤2 轮自动修复循环 |
| **工具链协同** | 7 个 LangChain Tool：搜索、计算、文件读写、数据分析、可视化、OCR |
| **结构化报告** | Summarizer Agent 汇总全流程，输出 Markdown 报告 + speaking_log 执行日志 |
| **Web 交互** | FastAPI + Jinja2 + daisyUI + Tailwind CSS 现代 Web 界面 |
| **数据持久化** | SQLite（WAL 模式）用户系统 + 会话历史 + 跨设备恢复 |

### 4.2 Router 意图分类器

`router.py` 是独立于 Agent 编排的轻量级分类器，通过 `urllib` 直接调用 DeepSeek API：

```
用户输入 → urllib POST → DeepSeek API
         → temperature=0, max_tokens=50, thinking=disabled
         → 解析返回的 "标签|复杂度"
         → 输出 (task_type, complexity)
```

**分类体系：**

| 标签 | 典型场景 | 复杂度判定 |
|------|---------|:----------:|
| 编程 | "写一个快排"、"实现 LRU Cache" | 重 |
| 写作 | "写一份市场分析报告" | 重 |
| 分析 | "分析 sales_data.csv" | 重 |
| 问答 | "什么是机器学习" | 轻 |
| 闲聊 | "你好"、"你是谁" | 轻 |

**关键词覆写 — 三层安全网（Router 后执行）：**

| 优先级 | 检测规则 | 触发词 | 覆写动作 |
|:------:|----------|--------|-----------|
| 1 | 搜索意图检测 | 搜索/查资料/检索/基于知识库 | complexity → 重（强制慢车道） |
| 2 | 分析意图检测 | 以"分析"开头 | complexity → 重 + task_type → 分析 |
| 3 | 非 Python 语言检测 | C语言/Java/Rust/Go/C++/C# | task_type → 问答 + complexity → 轻 |

### 4.3 七个 Agent 详解

| Agent | 图标 | 定位 | 绑定工具 | 核心职责 |
|-------|:----:|------|----------|----------|
| **Planner** | 📋 | 任务规划师 | 无 | 拆解复杂任务为步骤清单，输出 task_type 供路由使用 |
| **Bot** | 🤖 | 快车道接待 | 无 | 快车道直接回复用户，唯一面向用户的 Agent |
| **Retriever** | 🔍 | 知识检索员 | `search_knowledge` | 查询 ChromaDB 知识库，为后续 Agent 注入背景知识 |
| **Coder** | 💻 | 代码工程师 | `write_file`, `read_file`, `calculate`, `analyze_data`, `visualize_data` | Python 代码编写 + 数据分析 + 图表生成 |
| **Writer** | ✍️ | 文档撰稿人 | `write_file`, `read_file`, `search_knowledge` | 报告/方案/文章撰写，支持引用知识库 |
| **Tester** | ✅ | 质量审查员 | `read_file` | 代码质量评审 + 文档内容审查，输出 ✅/❌ 判定 |
| **Summarizer** | 📊 | 报告汇总师 | 无 | 汇总全流程执行记录，生成结构化 Markdown 报告 |

> **注：** Router 不属于 Agent，是独立轻量 LLM 调用（`router.py`，约 60 行）。所有 7 个 Agent 统一使用 DeepSeek V4 Flash，通过 LangChain `ChatDeepSeek` 接入。

### 4.4 LangGraph 工作流状态

```python
class WorkflowState(TypedDict):
    user_input: str                    # 用户原始输入
    lane_mode: str                     # "auto" | "fast" | "slow"
    task_type: str                     # "编程" | "写作" | "分析" | "问答" | "闲聊"
    complexity: str                    # "轻" | "重"
    plan: str                          # Planner 输出的步骤计划
    knowledge: str                     # Retriever 检索到的知识
    code_or_draft: str                 # Coder/Writer 产出的代码或文稿
    execution_result: str              # Executor 执行结果（含 exitcode）
    test_result: str                   # Tester 评审结果（含 ✅/❌）
    fix_count: int                     # 当前修复轮数（上限 2 轮）
    agent_messages: Annotated[list, operator.add]   # Agent 消息累积
    speaking_log: Annotated[list, operator.add]     # 发言流转日志
    final_output: str                  # 最终输出文本
```

**条件路由表（5 个关键分支点）：**

| 路由点 | 条件 | 目标 |
|--------|------|------|
| 入口分发 | `complexity == "轻"` | → Bot（快车道） |
| 入口分发 | `complexity == "重"` | → Planner（慢车道） |
| Retriever 后 | `task_type == "写作"` | → Writer |
| Retriever 后 | `task_type in ("编程","分析")` | → Coder |
| Executor 后 | `exitcode == 0` | → Tester（评审） |
| Executor 后 | `exitcode ≠ 0` | → 检查 fix_count，≤2 则 → Coder（修复），>2 则 → Summarizer |
| Tester 后 | 含 `✅` | → Summarizer（通过） |
| Tester 后 | 含 `❌` 且 `fix_count < 2` | → Coder/Writer（修复） |
| Tester 后 | `fix_count >= 2` | → Summarizer（强制结束） |

### 4.5 工具系统

7 个 LangChain Tool，统一使用 `@tool` 装饰器注册：

| 工具 | 类型 | 核心实现 | 功能描述 |
|------|------|----------|----------|
| `search_knowledge` | RAG 检索 | ChromaDB 向量查询 | 语义检索知识库，min_score=0.40 过滤低相关结果 |
| `read_file` | 文件读取 | `open().read()` | 读取 coding/ 目录下的生成文件 |
| `write_file` | 文件写入 | `open().write()` | 将代码/文稿写入 coding/ 目录 |
| `calculate` | 安全计算 | AST 白名单求值 | 数学表达式计算，仅允许安全运算符和内置函数 |
| `analyze_data` | 数据分析 | Pandas 分组聚合 | CSV/Excel 自动检测分组列和聚合列，输出统计结果 |
| `visualize_data` | 图表生成 | Pillow ImageDraw | 柱状图/折线图，支持 CJK 字体渲染 |
| `ocr_image` | 图片识别 | Tesseract | 中英文混合 OCR，识别结果可自动入库 |

### 4.6 用户与会话持久化

v3.2 新增 SQLite 数据层，使用 Python 标准库 `sqlite3`，WAL 模式 + 外键约束：

```
数据库：data.db（自动创建，已加入 .gitignore）

users 表：
  id          TEXT PRIMARY KEY    -- UUID
  name        TEXT UNIQUE         -- 用户名（幂等：重复注册返回同一 ID）
  created_at  TEXT                -- ISO 8601 时间戳

sessions 表：
  id          TEXT PRIMARY KEY    -- UUID
  user_id     TEXT                -- 外键 → users.id
  title       TEXT                -- 会话标题（首条消息截取）
  messages    TEXT                -- JSON 序列化的完整消息数组
  updated_at  TEXT                -- ISO 8601 时间戳
```

**设计要点：**
- **幂等用户注册**：同一用户名多次请求返回同一 user_id，前端 localStorage 缓存映射
- **自动保存**：每次对话完成后，前端自动 POST `/api/sessions` 保存全量消息
- **历史恢复**：侧边栏按时间倒序展示会话列表，点击即恢复完整对话上下文
- **连接安全**：context manager 管理连接生命周期，异常时自动 rollback

## 五、关键设计决策

### 5.1 决策总览

| # | 决策点 | 选择 | 核心理由 |
|---|--------|------|----------|
| 1 | 多 Agent 框架 | **LangGraph** 替代 AG2 | 显式 StateGraph，可调试可扩展，社区活跃 |
| 2 | 前端框架 | **FastAPI Web** 替代 Streamlit | 标准 HTTP，完全 UI 控制，可融入 ASGI 生态 |
| 3 | 意图分类 | **Router LLM + 关键词覆写** | 自动化覆盖 85% + 安全网兜底 15% |
| 4 | 执行结果判断 | **exitcode 确定性分支** | 操作系统返回值，不受 LLM 幻觉影响 |
| 5 | CSS 框架 | **daisyUI 5** 替代 Bootstrap 5 | 语义组件更丰富，drawer/collapse/join 零 JS |
| 6 | 数据持久化 | **SQLite WAL** 替代 localStorage | 跨设备、多用户、数据不丢失、支持并发 |
| 7 | 应用生命周期 | **lifespan** 替代 on_event | 避免 FastAPI 弃用警告，资源管理更清晰 |

### 5.2 决策 1：AG2 → LangGraph

**AG2 在 v2.x 中暴露的问题：**
- 黑盒调度：Agent 间消息传递不透明，调试困难
- 串行限制：无法灵活控制并行/串行切换
- 社区迁移：AG2 维护活跃度下降，LangGraph 成为 LangChain 生态标准

**LangGraph 带来的改善：**
- **显式状态图**：`StateGraph` + `add_conditional_edges`，工作流即文档
- **确定性路由**：`exitcode == 0` 这种条件直接写在代码里，不是 prompt 里
- **状态累积**：`Annotated[list, operator.add]` 自动聚合多节点输出
- **生态整合**：与 LangChain ChatDeepSeek、@tool 装饰器无缝配合

### 5.3 决策 2：Streamlit → FastAPI Web

**Streamlit 的实际限制（v3.0 亲身经历）：**

| 问题 | 影响 |
|------|------|
| `st.rerun()` 全页刷新 | 无法做精细 DOM 操作，用户体验割裂 |
| 组件不可定制 | `st.chat_message`、`st.expander` 样式固定 |
| 部署耦合 | 只能 `streamlit run`，无法融入 Nginx/Gunicorn 体系 |
| 纯 SPA 渲染 | SEO 不可行，无法被搜索引擎索引 |

**FastAPI Web 的改善：**
- 标准 HTTP：GET/POST，可用 curl、Postman、任何 HTTP 客户端测试
- 完全 UI 控制：HTML/CSS/JS 自由定制，不受框架组件约束
- ASGI 生态：可部署到 Uvicorn、Gunicorn、Daphne 等标准服务器
- 渐进升级：当前 Jinja2 渲染，未来可平滑迁移到 React/Vue 前端

### 5.4 决策 3：exitcode 驱动分支 vs LLM 文字判断

v3.0 中 Executor 执行代码后，将 stdout/stderr 纯文本传给 Tester 做文字判断。**问题：** Tester 是 LLM，可能误判——代码已经跑通了但 Tester 说"看起来有问题"。

v3.1 加入 exitcode 驱动分支：

```
executor 执行代码
  │
  ├─ exitcode = 0  →  代码执行成功  →  Tester 评审
  │
  └─ exitcode ≠ 0  →  代码执行失败  →  检查 fix_count
                                         ├─ ≤ 2  →  Coder 修复（传入 stderr）
                                         └─ > 2  →  Summarizer（放弃修复）
```

**为什么这很重要：** exitcode 是操作系统内核返回的确定性值，不受 LLM 幻觉影响。能用代码判断的，就不要问 LLM。

### 5.5 决策 4：localStorage → SQLite 持久化

v3.1 的会话数据仅存浏览器 localStorage，存在三个致命问题：

| 问题 | 影响 |
|------|------|
| 无法跨设备 | 换浏览器/设备，历史全部丢失 |
| 无法多用户 | 同一浏览器只能有一个用户的上下文 |
| 数据不可靠 | 清理浏览器数据即丢失，无法备份 |

选择标准库 `sqlite3` 而非 ORM 的理由：
- **零额外依赖**：Python 标准库自带，不需要安装 SQLAlchemy 等
- **WAL 模式**：Write-Ahead Logging 支持读写并发，页面刷新不阻塞
- **context manager**：自动 commit/rollback/close，异常安全

## 七、v3.2 核心亮点

### 7.1 daisyUI + Tailwind CSS 前端重构

v3.1 使用 Bootstrap 5 CDN，组件定制能力有限。v3.2 全面迁移到 daisyUI 5 + Tailwind CSS 4：

| 按钮组 | 手动 flex 布局 | **join** 语义组件 | 自动圆角整合，视觉更紧凑 |
| 弹窗 | Bootstrap modal（JS 依赖） | **modal + dialog** | 原生 `<dialog>` 元素兜底 |

| 加载态 | 手动 spinner | **loading 工具类** | 语义化，任意元素可加 |

迁移后 HTML 代码量减少约 30%，组件语义化程度显著提升。

### 7.2 SQLite 用户与会话持久化

从零构建完整数据持久化层（`db.py`，约 160 行）：

- **WAL 模式**：写操作不阻塞读操作，支持多 tab 并发访问
- **外键约束**：`sessions.user_id → users.id`，保证引用完整性
- **幂等用户注册**：同一用户名多次注册返回同一 user_id，前后端双重缓存
- **自动保存 + 历史恢复**：对话结束自动持久化，侧边栏按时间分组展示历史

### 7.3 lifespan 应用生命周期管理

```python
# v3.1（已弃用）
@app.on_event("startup")
async def startup():
    app.state.db = Database("data.db")

# v3.2（标准方式）
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.db = Database(os.path.join(_PROJECT_DIR, "data.db"))
    yield  # 应用运行期间
    # yield 之后自动清理资源
```

### 9.2 关键收获

1. **LangGraph 显式图 > 黑盒调度**：`add_conditional_edges` 让工作流逻辑一目了然，加分支只需一行代码，调试时每个节点的输入输出都可检查
2. **自动 + 安全网 = 最优解**：Router 自动分类覆盖 85%，关键词正则兜底 15%，纯手动或纯自动都不如两者结合
3. **确定性信号 > LLM 判断**：exitcode 是操作系统返回值，比 LLM 的"看起来没问题"可靠得多——能用代码判断的绝不问 LLM
4. **标准 Web > 框架锁定**：FastAPI + Jinja2 + daisyUI 是 Python Web 的现代组合，任何 Python 开发者都能理解和维护
5. **持久化是产品的分水岭**：SQLite + WAL 让系统从"演示 Demo"升级为"可用的产品"，用户数据不再丢失，跨设备可访问
6. **daisyUI 语义化优势**：drawer/collapse/join 等语义组件让 HTML 结构即文档，比通用 div+class 更易维护

### 9.3 后续可改进方向

| 方向 | 描述 | 优先级 |
|------|------|:------:|
| **WebSocket 流式推送** | 当前一次性返回全流程结果，改为 SSE/WS 逐 Agent 推送可大幅提升感知速度 | 高 |
| **Docker 沙箱隔离** | subprocess 在宿主机执行，升级为 Docker 容器隔离更安全 | 高 |
| **并行 Agent 执行** | LangGraph 支持 `Send()` API，可将规划/检索等独立任务并行化 | 中 |
| **前端渐进升级** | 当前原生 JS 可平滑迁移到 htmx 或 Alpine.js，保持零构建工具 | 中 |
| **A/B 模型对比** | 同一任务并行跑两个不同模型，对比输出质量 | 低 |
| **长期记忆层** | 当前仅注入最近 3 轮摘要，可升级为 Mem0 或 LangChain Memory | 低 |
