## 一、项目背景与目标

### 1.1 起点：实训 Day 10 的四角色协作

实训 Day 10 完成了 `10_05_1多智能体/`——一个基于 AG2 的四角色协作系统：

| 角色 | 模型 | 职责 |
|------|------|------|
| Coder | Qwen 7B | 编写代码 |
| Tester | DeepSeek Flash | 审查代码 |
| Optimizer | DeepSeek Flash | 修复代码 |
| User | — | 执行代码 |

该系统使用自定义状态机（`speaker_selection`）控制发言顺序，`exitcode` 驱动流程分支，最多 2 轮迭代修复。协作任务是"带过期时间的 LRU Cache"。

**局限：**
- 仅支持编程任务，缺少写作/分析/闲聊能力
- 控制台交互，无可视化界面
- 无知识库，所有知识依赖 LLM 自身
- 4 角色 pipeline 对轻量任务（如"你好"）过度启动全部链路

### 1.2 2.0 目标

| 维度 | 1.0 | 2.0 |
|------|-----|-----|
| 角色数 | 4 | 8 AG2 Agent + 1 Router |
| 任务类型 | 仅编程 | 编程/写作/分析/问答/闲聊 |
| 前端 | 无（控制台）| Streamlit 聊天界面 |
| 知识库 | 无 | ChromaDB + bge 嵌入 |
| 轻量任务 | 走全链路 | 快车道 Bot 直回 |
| 报告 | 无（手动）| 按钮触发 Summarizer 生成 |
| 配置 | 硬编码 | MODEL_POOL + ROLE_MODEL 可插拔 |

### 2.2 实体清单（8 Agent + 1 Router）

| 类型 | 名称 | 实现方式 | 模型 | 职责 |
|------|------|---------|------|------|
| 路由 | **Router** | 直接 HTTP API 调用 | DeepSeek Chat | 意图分类 + 复杂度判定 |
| Agent | **Planner** | AG2 AssistantAgent | DeepSeek Chat | 拆解任务 → 步骤清单 |
| Agent | **Bot** | AG2 AssistantAgent | DeepSeek Chat | 快车道响应，唯一面客 |
| Agent | **Retriever** | AG2 AssistantAgent | Qwen 7B | 查询 ChromaDB 知识库 |
| Agent | **Coder** | AG2 AssistantAgent | Qwen 7B | 编写 Python 代码+测试 |
| Agent | **Writer** | AG2 AssistantAgent | DeepSeek Chat | 撰写报告/方案/文章 |
| Agent | **Tester** | AG2 AssistantAgent | DeepSeek Chat | 通用评审（代码+文档）|
| Agent | **Summarizer** | AG2 AssistantAgent | DeepSeek Chat | 汇总执行过程 → 结构化报告 |
| Agent | **User** | AG2 UserProxyAgent | — | 执行 Python 代码 |

### 2.3 Router 意图分类器（`router.py`）

Router 不通过 AG2 agent 实现，而是直接构造 HTTP 请求调用 LLM API。原因：
- AG2 agent 会被视为 GroupChat 参与者（污染状态机路由）
- 直接 API 调用更快（<0.5s）、更便宜、更可靠
- 分类失败时默认降级为 `(闲聊, 轻)`

**分类维度：**
- **标签**：编程 / 写作 / 分析 / 问答 / 闲聊
- **复杂度**：轻（示例/告知型/概念解释）/ 重（完整项目/复杂算法/数据分析）

**分类逻辑：**
```
"写一个最简单的c程序"  → 编程 | 轻
"实现一个完整的LRU缓存" → 编程 | 重
"什么是机器学习"       → 问答 | 轻
"分析这个CSV数据"      → 分析 | 重
"你好"                → 闲聊 | 轻
```

### 2.4 快慢车道分离（`app/chat.py`）

```
复杂度「轻」→ _run_fast() → Bot.generate_reply() → 直接返回（无 thinking card）

复杂度「重」→ _run_slow() → GroupChat 多 Agent pipeline
                              │
              ┌───────────────┴──────────────┐
              │ 编程/分析 → coding lane       │ 写作 → writing lane
              │ max_round=12                  │ max_round=10
              └──────────────────────────────┘
```

**非 Python 语言检测（第二层防护）：**
```python
# 正则拦截 C/Java/Rust/Go/Swift/C++/C#/TypeScript
_non_py = re.search(
    r'(c语言|c\s*代码|java|rust|go\s*语言|golang|swift|c\+\+|c#|typescript)',
    user_input, re.IGNORECASE
)
if _non_py and task_type == "编程":
    task_type = "问答"
    complexity = "轻"   # 强制走 Bot 快车道
```

### 2.5 双 Lane 状态机（`groupchat.py`）

**coding lane（编程/分析）：**
```
User → Planner → Retriever → Coder → Tester ⇄ Coder（≤2轮）
                                                    ↓ ✅
                                                  User（执行代码）
                                                    ↓
                                exitcode=0 → Summarizer → 结束
                                exitcode≠0 → Coder（≤2轮重试）
                                exitcode超限 → Summarizer
```

**writing lane（写作/策划）：**
```
User → Planner → Retriever → Writer → Tester ⇄ Writer（≤2轮）
                                                ↓ ✅
                                            Summarizer → 结束
```

**状态机规则要点：**
- Tester 回复含 `✅` → 核准通过 → 进入下一阶段
- Tester 回复含 `❌` → 退回 Coder/Writer 修复
- 修复迭代最多 `_MAX_FIX_CYCLES = 2` 轮
- User 的 `exitcode` 驱动 coding lane 的关键分支
- 所有发言顺序记录在 `speaking_log` 中

## 三、关键设计决策

### 3.1 决策总览表

| # | 决策 | 为什么 | 权衡 |
|---|------|--------|------|
| 1 | **Router 不进入 GroupChat** | AG2 agent 会被视为发言者（污染状态机）；直接 API 调 < 0.5s，更便宜 | 需额外维护 HTTP 请求代码 |
| 2 | **快慢车道分离** | "你好"/"什么是 Python" 等不需走 6 agent 链；复杂度分级节省约 50% API 调用 | Router 误判会导致任务降级 |
| 3 | **弱写强查策略** | Coder/Retriever 用免费 Qwen 7B（本地 Ollama），Planner/Tester/Summarizer 用 DeepSeek Chat | 弱模型代码质量靠强 Tester 纠错 |
| 4 | **双 Lane 自定义状态机** | LLM 自主选择发言者 (`auto` 模式) 不可控；exitcode 驱动比自然语言判断更可靠 | 需维护 coding + writing 两套逻辑 |
| 5 | **非 Python 语言正则拦截** | Router 分类时无法识别语言；C/Java/Rust 代码在 coding lane 无法执行 → 无限重试 → 超时 | 正则误报可能错误拦截（概率低）|
| 6 | **软超时 90s（非硬 kill）** | 硬 kill 线程丢失所有中间结果；软超时收集已产出的 Agent 消息返回 | 后台线程继续运行消耗资源 |
| 7 | **thinking card 默认折叠** | 答辩/展示时，评委只关心最终结果；有兴趣才展开查看内部协作过程 | 需额外前端组件 |
| 8 | **报告按需生成（按钮触发）** | 避免每轮自动生成浪费 API；用户觉得有价值时才点击 | 多一步操作 |
| 9 | **MODEL_POOL + ROLE_MODEL 分离** | 不同设备只需改 config.py；定义和引用解耦；team members 易于协作 | 配置复杂度略增 |

### 3.2 决策 1 详解：Router 独立设计

**方案 A（放弃）：** 用 AG2 AssistantAgent 做分类
- 问题：Agent 进入 GroupChat 发言列表，状态机的 `last_speaker` 被污染
- 问题：Router 的分类响应会被 `_filter_thinking` 误认为思考记录
- 问题：每次分类多消耗一个 AG2 对话轮次

**方案 B（采纳）：** 直接构造 urllib HTTP 请求
- 读取 `config.py` 的模型配置（复用 Planner 的模型）
- `POST /chat/completions`，temperature=0，max_tokens=20
- 解析返回的 `标签|复杂度` 格式（如 `编程|重`）
- 异常兜底：任何错误 → `(闲聊, 轻)`

```python
# router.py 核心逻辑
def classify(user_input: str) -> tuple[str, str]:
    info = _get_model_info()
    payload = {...}           # temperature=0, max_tokens=20
    req = Request(url, data=payload, headers, method="POST")
    resp = urlopen(req, timeout=15)
    raw = data["choices"][0]["message"]["content"]  # "编程|重"
    parts = raw.split("|")
    # 解析得到 (task_type, complexity)
```

### 3.3 决策 5 详解：非 Python 语言的三层防护

| 防护层 | 位置 | 机制 | 作用 |
|--------|------|------|------|
| 第 1 层 | `router.py` prompt | "HelloWorld/C 程序 → 轻 + 问答" | Router 本身有常识判断 |
| 第 2 层 | `app/chat.py` 正则 | `re.search(r'c语言|java|rust|...')` | 拦截 Router 漏判 |
| 第 3 层 | `agents.py` Coder prompt | "只能写 Python，非 Python 标注提示" | 最终兜底 |

### 3.4 决策 6 详解：软超时 vs 硬超时

**硬超时方案（放弃）：**
```python
t.join(timeout=60)
if t.is_alive():
    return {"reply": "超时，已中断。", "thinking": []}
```
- 问题：线程跑了几步，Coder 已生成代码，Tester 已审查——全部丢弃
- 用户得到空结果，体验差

**软超时方案（采纳）：**
```python
t.join(timeout=90)
partial_msgs = list(mgr.groupchat.messages)  # 线程未 kill，继续写入
if t.is_alive():
    thinking = _filter_thinking(partial_msgs)
    if thinking:
        return {"reply": f"已收集 {len(thinking)} 条中间结果", "thinking": thinking}
```
- daemon 线程继续后台跑，不 kill
- 已产出的 Agent 消息全量返回
- 用户能看到走到哪一步了

### v2.0 阶段修复（2026-06-11 → 16）

从四角色扩展到八角色 + Router 的过程中，主要遇到两类问题：**环境配置类**（#1-#3）和 **AG2 框架理解类**（#4-#6）。其中代理和镜像问题属于 Windows 开发环境的典型坑，而 function_map 注册范围和 thinking 模式冲突则需要对 AG2 内部机制有深入理解才能解决。

### 问题 1：系统代理导致 Ollama 502

**现象：** Qwen 7B 推理阶段报 `httpx.InternalServerError: Error code: 502`

**排查过程：**
1. `ollama list` 正常，`curl http://localhost:11434` 正常
2. Python 脚本中 `requests.get("http://localhost:11434")` 返回 502
3. `echo $env:HTTP_PROXY` → `http://127.0.0.1:7897`（Clash 代理）

**根因：** 系统级 HTTP 代理拦截了到 `localhost:11434` 的请求

**修复：** 在 `main.py` 和 `agents.py` 头部设置绕过规则
```python
os.environ["NO_PROXY"] = "localhost,127.0.0.1"
os.environ["no_proxy"] = "localhost,127.0.0.1"
```
注意：`NO_PROXY`（大写）和 `no_proxy`（小写）需同时设置，不同库读取不同的变量名。

---

### 问题 2：HuggingFace 直连超时

**现象：** 首次运行 `BAAI/bge-small-zh-v1.5` 下载卡在 0%，最终 `OSError: [WinError 10060]`

**根因：** HuggingFace CDN 在国内被墙

**修复：** 使用国内镜像
```python
os.environ["HF_ENDPOINT"] = "https://hf-mirror.com"
```
需在两处设置：`main.py`（Streamlit 进程）和 `rag/knowledge_base.py`（子模块独立导入时）

---

### 问题 3：stdout 重定向导致 tqdm OSError

**现象：** 重建知识库索引时 `OSError: [Errno 9] 文件描述符有误`，发生在 tqdm 进度条

**排查：** AG2 框架默认在 `coding/` 目录下执行代码时会将 `sys.stdout` 替换为文件对象，Streamlit 环境中该文件描述符无效

**修复：** 移除 `main.py` 中的 `sys.stdout = open(...)` 替换代码，Streamlit 自行处理编码问题

---

### 问题 4：工具函数 "not found"

**现象：** Retriever 调用 `search_knowledge` 返回 `Error: Function search_knowledge not found`

**排查过程：**
1. 确认 `register_function(fn, caller=retriever, executor=user)` 注册成功
2. 确认 Retriever 的 prompt 中有工具说明
3. 打印 `retriever._function_map` → 函数已注册
4. 观察 call stack → 函数是在**接收方代理**执行，而非调用方

**根因：** AG2 GroupChat 的 `execute_function` 机制——当 Agent A 回复"建议调用函数 F"，Agent B（下一发言者）在准备发言前会尝试执行 F。但 `_function_map` 仅注册在 Retriever 和 User，其他 Agent 找不到。

**修复：** 将 `_function_map` 注册到全部 8 个 Agent + 2 个 Manager
```python
for agent in _all_agents:  # 8 个 Agent
    agent._function_map[fn.__name__] = fn

for mgr in (coding_manager, writing_manager):  # 2 个 Manager
    mgr._function_map[fn.__name__] = fn
```

### 问题 5：DeepSeek thinking 模式 400 错误

**现象：** `openai.BadRequestError: 400 — The reasoning_content field is only supported in requests and must be passed back to the API`

**排查过程：**
1. 错误仅发生在 DeepSeek API（非 Qwen 7B），排除网络问题
2. 检查请求 payload → 发现 DeepSeek 返回的 `reasoning_content` 字段
3. AG2 框架构造新一轮请求时未保留 `reasoning_content`（OpenAI 标准不含该字段）
4. DeepSeek 要求：如果请求包含 `reasoning_content`，后续请求必须回传

**修复（两层）：**

第一层：换模型名
```python
# config.py — 从 deepseek-v4-pro 改为 deepseek-chat（无 thinking 模式）
MODEL_POOL = {
    "a-deepseek": {"model": "deepseek-chat", ...},
}
```

第二层：OpenAI create 方法 patch（双重保险）
```python
# main.py
_orig_create = openai.resources.chat.completions.Completions.create
def _patched_create(self, **kwargs):
    if "deepseek" in str(kwargs.get("model", "")).lower():
        kwargs.setdefault("extra_body", {})
        kwargs["extra_body"]["thinking"] = {"type": "disabled"}
    return _orig_create(self, **kwargs)
openai.resources.chat.completions.Completions.create = _patched_create
```

### 问题 6：NoneType.replace 崩溃

**现象：** `'NoneType' object has no attribute 'replace'`，发生在 Tester→User 路由后

**排查过程：**
1. 错误来自 AG2 框架内部：UserProxyAgent 提取代码块时对 content 调 `.replace()`
2. Tester 的响应 content 可能为 `None`（当 Tester 仅发出工具调用时）

**修复：** 修改 `groupchat.py` 状态机——Tester 仅在回复包含 `✅` 时才路由到 User
```python
if name == "Tester":
    if "✅" in content:          # 只有明确核准才继续
        if _has_code_in_conversation(msgs):
            return _next(last_speaker, agent_map.get("User"))
        return _next(last_speaker, agent_map.get("Summarizer"))
    # 否则退回 Coder 修复
```

同时在 `main.py` 的 OpenAI patch 中增加 null guard：
```python
for tc in (m.get("tool_calls") or []):
    fn = tc.get("function", {})
    if fn.get("arguments") is None:
        fn["arguments"] = "{}"    # 防止 AG2 _format_json_str(None)
```

### 问题 7：C 语言请求走慢车道卡死

**现象：** 用户输入"写一个完整的 C 语言程序"，Router 分类为 `编程 | 重`，进入 coding lane → Coder 写 Python → Tester 通过 → User 无法编译 C → exitcode ≠ 0 → 不断重试 → 60s 超时

**根因分析链：**
1. Router 的 system prompt 未覆盖"非 Python 编程语言"场景
2. Router 将"C 语言程序"分类为 `编程 | 重`（合理，因为是"编程"）
3. Coder 限定 Python only，但遇到 C 任务只能出代码片段
4. User 执行器仅支持 Python，遇到 C 代码 exitcode ≠ 0
5. 状态机将 exitcode ≠ 0 路由回 Coder → 无限循环 → 超时

**修复（两层）：**

第一层：语言检测正则（`app/chat.py`）
```python
_non_py = re.search(
    r'(c语言|c\s*代码|java|rust|go\s*语言|golang|swift|c\+\+|c#|typescript)',
    user_input, re.IGNORECASE
)
if _non_py and task_type == "编程":
    task_type = "问答"
    complexity = "轻"     # 强制走 Bot 快车道
```

第二层：软超时 90s（防止类似漏网）
```python
_TIMEOUT_SLOW = 90  # 从 60s 延长
t.join(timeout=90)
if t.is_alive():
    partial_msgs = list(mgr.groupchat.messages)  # 不 kill，收集已有
    thinking = _filter_thinking(partial_msgs)
    if thinking:
        return {"reply": f"已收集 {len(thinking)} 条中间结果", ...}
```

**验证：**
- "写一个最简单的c程序" → Router: `编程|重` → 正则拦截 → `问答|轻` → Bot 秒回 C 代码
- "实现一个LRU缓存" → Router: `编程|重` → 正则不匹配 → 正常走 coding lane

### 问题 8：GroupChat 无限循环

**现象：** 编码任务中，Tester 反复驳回 Coder，Coder 反复修复，超出预期轮次

**根因：** `max_round=18`（coding）/ `max_round=14`（writing）太大，允许过多轮次

**修复：**
```python
coding_groupchat = GroupChat(agents=..., max_round=12, ...)   # 18 → 12
writing_groupchat = GroupChat(agents=..., max_round=10, ...)  # 14 → 10
```

仍保留 2 轮 Coder↔Tester + 2 轮 User→Coder 的修复余量（6 固定轮次 + 6 弹性轮次）。

### 2.1 UI 改造记录（2026-06-17）

v2.1 主要聚焦 Streamlit 前端的用户体验改进，涉及多个 UI 组件的设计迭代：

**1. 思考过程（thinking card）UI 设计**
- 初版用 `st.checkbox` 切换折叠状态 + 底部固定收起按钮
- 问题：`st.checkbox` 点击后整个组件坍缩，内容短时仍占用空间
- 最终方案：改用 `st.expander` + 独立滚动容器（`max-height: 400px; overflow-y: auto`），统一使用 `st.markdown` 渲染，收起/展开按钮始终可见

**2. 用户消息 markdown 转义**
- 问题：用户输入中的 `#`、`**`、`` ` `` 等字符被 st.markdown 错误渲染为标题/加粗
- 修复：添加 `escape_md()` 函数，对 `# * _ ~ [ ] ( )` 等特殊字符转义

**3. 右对齐 CSS 尝试**
- 尝试将用户消息右对齐以区分对话双方
- 导致：消息容器错乱、滚动位置异常
- 结果：放弃，恢复 Streamlit 默认左对齐

**4. 侧边栏对话目录**
- 新增最近 10 条对话的快速跳转链接
- 关键修复：嵌套页面（Streamlit iframe）中 JS 必须使用 `window.parent.document.getElementById` 而非 `document.getElementById`

**5. 搜索任务分流调整**
- 搜索关键词命中后从 coding lane 改为 writing lane，避免触发 Coder 写代码空转超时
- 搜索检测优先级提到非 Python 语言降级之上

### 7.2 关键收获

1. **AG2 框架内部机制**：`register_function` 的执行时机、`speaker_selection_method` 的自定义、`_function_map` 的注册范围，都需深入框架源码才能理解
2. **Router 独立设计**：不把 Router 做成 Agent 是正确的——速度快、逻辑清晰、不污染状态机
3. **弱写强查的成本优化**：Coder 用免费 Qwen 7B 生成代码，强 Tester（DeepSeek）来审查——相当于"便宜写、贵审"，整体成本降低约 40%
4. **超时策略的权衡**：软超时比硬超时更友好——即使超时用户也能看到部分结果
5. **前端体验设计**：thinking card 默认折叠、报告按需生成——适合答辩场景（评委只看结果）

### 7.3 可改进方向

- 用 LangGraph 替代自定义 if/else 状态机，获得可视化流程图和更好的状态管理
- Docker 沙箱执行非 Python 代码（C/Java/Rust），扩展 Coder 的能力边界
- 流式输出 thinking content，用户可实时看到 Agent 工作过程
- 多轮对话的长期记忆（当前仅注入最近 3 轮摘要）
- 多模态支持（图片/语音输入）
