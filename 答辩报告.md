### 1.2 v3.0 目标

| 维度 | v2.x | v3.0 |
|------|------|------|
| Agent 框架 | AG2 (AutoGen) | **LangGraph + LangChain** |

| 启动速度 | ~40-60s | **~10-15s（首次）/ <5s（热启动）** |

### 2.2 Agent 清单（7 个）

| Agent | 图标 | LLM | 工具 | 职责 |
|-------|:----:|-----|------|------|
| **Planner** | 📋 | DeepSeek V4 Flash | 无 | 拆解任务为步骤清单，判断 task_type |
| **Bot** | 🤖 | DeepSeek V4 Flash | 无 | 快车道直接回复，唯一面客 |
| **Retriever** | 🔍 | DeepSeek V4 Flash | `search_knowledge` | 查询 ChromaDB 知识库 |
| **Coder** | 💻 | DeepSeek V4 Flash | `write_file`, `read_file`, `calculate` | 编写 Python 代码 |
| **Writer** | ✍️ | DeepSeek V4 Flash | `write_file`, `read_file`, `search_knowledge` | 撰写报告/方案/文章 |
| **Tester** | ✅ | DeepSeek V4 Flash | `read_file` | 通用评审（代码+文档） |
| **Summarizer** | 📊 | DeepSeek V4 Flash | 无 | 汇总执行过程 → 结构化报告 |

> v3.0 移除了 v2.x 中依赖本地 Ollama 的 Qwen 7B 模型，所有 7 个 Agent 统一使用 DeepSeek API。代码执行不再需要独立的 User Agent，由 `executor.py` 沙箱直接处理。

### 2.3 LangGraph 工作流

**状态定义：**

```python
class WorkflowState(TypedDict):
    user_input: str
    lane_mode: str          # "fast" | "slow"
    task_type: str          # "coding" | "writing"
    plan: str
    knowledge: str
    code_or_draft: str
    execution_result: str
    test_result: str
    fix_count: int          # 当前修复轮数 (max 2)
    agent_messages: list    # 所有中间消息 (streaming 给前端)
    final_output: str
```

**条件路由：**

| 路由点 | 条件 | 结果 |
|--------|------|------|
| 入口 | `lane_mode == "fast"` | → Bot |
| 入口 | `lane_mode == "slow"` | → Planner |
| Retriever 后 | `task_type == "coding"` | → Coder |
| Retriever 后 | `task_type == "writing"` | → Writer |
| Tester 后 | `"✅" in test_result` | → Summarizer |
| Tester 后 | `"❌" in test_result and fix_count < 2` | → Coder/Writer（修复） |
| Tester 后 | `fix_count >= 2` | → Summarizer（超限强制结束） |

**慢车道流水线：**

```
Planner → Retriever → [Coder 分支] → Executor → Tester → Summarizer
                    → [Writer 分支] → Tester → Summarizer
                                     ↑    │
                                     └────┘ (≤2轮修复循环)
```

### 2.4 快慢车道切换

v2.x 使用 `router.py` 自动分类——Router 调用 LLM 判断任务复杂度后自动选择车道。**实践中 Router 误判率约 15%**（如"分析"关键词误触发慢车道导致简单分析走全链路）。

v3.0 改为 **侧边栏手动选择**——用户在 `st.radio` 中显式选择快/慢车道，系统强制执行：

- **快车道**：所有输入 → Bot 直接回复，不走任何流水线
- **慢车道**：所有输入 → 完整 7 Agent 流水线

切换即时生效，读取 `st.session_state.lane_mode`。

### 2.5 工具系统

7 个 LangChain Tool，使用 `@tool` 装饰器定义：

| 工具 | 类型 | 功能 |
|------|------|------|
| `search_knowledge` | RAG | ChromaDB 向量检索（min_score=0.40） |
| `read_file` | 文件 | 读取 coding/ 目录文件 |
| `write_file` | 文件 | 写入 coding/ 目录文件 |
| `calculate` | 计算 | AST 安全数学表达式求值 |
| `analyze_data` | 数据 | Pandas 分组聚合分析 |
| `visualize_data` | 可视化 | **Pillow** 绑图（柱状图/折线图） |
| `ocr_image` | OCR ★新增 | **Tesseract** 中英文图片文字识别 |

## 三、关键设计决策

### 3.1 决策总览

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| 1 | **AG2 → LangGraph** | 全面迁移 | 显式图 > 手写 if/else；检查点恢复；原生 streaming |
| 2 | **Router 移除 → 手动车道** | 侧边栏 radio | 消除 Router 15% 误判率；用户自主控制更可靠 |
| 3 | **Qwen 7B 移除** | 全部 DeepSeek API | Ollama 本地推理慢且不稳定；统一 API 简化维护 |
| 4 | **代码执行沙箱化** | subprocess 隔离 | 替代 AG2 UserProxyAgent；超时控制 + 自动清理 |
| 5 | **嵌入模型离线加载** | HF_HUB_OFFLINE=1 | 避免每次启动检查 HuggingFace 在线状态（网络超时 ~30s） |
| 6 | **Tesseract OCR 集成** | 知识库入库 + 对话识别 | 图片→文字→RAG，扩展文档输入形式 |
| 7 | **UI 深蓝主题** | 自定义 CSS | 专业仪表盘风格，区别于默认 Streamlit 样式 |

### 3.2 决策 1 详解：为什么从 AG2 迁移到 LangGraph

**AG2 的问题（v2.x 实际遇到的）：**

1. **状态机隐式化**：`speaker_selection_method` 中的 if/else 链在 250 行后变得难以追踪——"谁在什么条件下到谁"分散在多处
2. **无检查点**：90s 超时时线程被收集但中间状态丢失——因为 AG2 GroupChat 没有持久化机制
3. **调试困难**：当 Tester→Coder 循环异常时，需要在 `speaking_log` 中手工追踪 13 步发言链路
4. **streaming 缺失**：AG2 GroupChat 的 `run()` 是阻塞的——前端只能显示"思考中..."直到全部完成

**LangGraph 的改善：**

1. **显式图**：节点 + 条件边一目了然。添加新 Agent = 加一个 `add_node` + 一个 `add_edge`
2. **StateGraph 状态可序列化**：未来可实现崩溃恢复
3. **节点级 tracing**：每个节点的输入/输出自动记录
4. **原生 streaming**：`astream_events()` 支持逐节点推送，前端可实时渲染每个 Agent 的输出

**为什么不保留 AG2 Agent + LangGraph 编排：**
- 两个框架的 Agent 模型不兼容（AG2 的 `generate_reply` vs LangChain 的 `invoke`）
- 维护两套依赖增加脆弱性
- 全部切 LangChain 后技术栈统一，RAG/工具/Agent 全链一致

### 3.3 决策 2 详解：手动车道 vs 自动 Router

| 维度 | Router 自动（v2.x） | 手动选择（v3.0） |
|------|-------------------|-----------------|
| 误判率 | ~15% | 0%（用户自主） |
| 延迟 | +0.5~2s（LLM 分类） | 0 |
| API 消耗 | 每次对话多 1 次 LLM 调用 | 0 |
| 用户体验 | 系统"猜"用户意图 | 用户明确控制 |

**验证案例（v2.x Router 误判）：**
- "写一个最简单的 c 程序" → Router 判 `编程|重` → 正则拦截才降级
- "分析一下这个字符串'hello world'" → Router 判 `分析|重` → 触发完整数据分析流水线，实际上用户只是问字符串含义

v3.0 的手动选择完全消除了这类误判。

## 四、技术栈

| 领域 | 技术 | 版本/型号 | 用途 |
|------|------|----------|------|
| Agent 框架 | **LangGraph** | ≥0.2 | 显式 StateGraph 编排 7 Agent 工作流 |
| Agent 实现 | **LangChain** | ≥0.3 | create_agent + @tool + ChatDeepSeek |
| LLM | DeepSeek API | deepseek-v4-flash | 全部 7 个 Agent |
| 前端 | Streamlit | ≥1.35 | 聊天界面 + 侧边栏 + 深蓝主题 CSS |
| 向量数据库 | ChromaDB | ≥1.5 | RAG 知识库存储与检索 |
| 嵌入模型 | BAAI/bge-small-zh-v1.5 | 33MB | 文档向量化（中文优化，离线加载） |
| 文档解析 | LangChain PyPDF + TextSplitter | — | PDF/TXT → chunk → 向量 |
| OCR | **Tesseract + Pillow** | ≥0.3.10 / ≥10 | 图片文字识别 + 图表绑定 |
| 数据分析 | Pandas | ≥2.0 | CSV/Excel 处理 |
| 代码执行 | subprocess | stdlib | 隔离沙箱执行用户代码 |

## 七、总结与展望

### 7.1 项目成果

| 指标 | v2.x | v3.0 |
|------|------|------|
| Python 文件 | 12 | **15** |
| Agent 数量 | 8 (+ Router) | **7**（统一模型） |
| 编排方式 | 手写状态机 | **LangGraph StateGraph** |
| 工具数量 | 5 | **7**（+visualize_data +ocr_image） |
| 启动时间 | ~40-60s | **~10-15s** |
| 支持模型 | DeepSeek + Qwen 7B | **DeepSeek V4 Flash** |
| Router 误判率 | ~15% | **0%**（手动选择） |

### 7.2 关键收获

1. **显式图 > 隐式状态机**：LangGraph 的 StateGraph 让工作流可读、可调试、可扩展
2. **自动化不总是最优**：Router 自动分类的 15% 误判率告诉我们——有些决策交给用户更可靠
3. **依赖统一降低维护成本**：去掉 Ollama/Qwen 后，环境配置从"装两个模型"简化为"设一个环境变量"
4. **离线加载的重要性**：HuggingFace 的网络超时在 Windows + 国内网络环境下是常态——强制离线是必要的启动优化
5. **LangChain Tool 的简洁性**：`@tool` 装饰器比 AG2 的 `register_function` 更直观，一个函数 = 一个工具

### 7.3 可改进方向

- **Docker 沙箱**：当前 subprocess 在宿主机直接执行，安全隔离不够
- **多模态输入**：OCR 已支持图片文字，扩展到图表理解、公式识别
- **长期记忆**：当前仅注入最近 3 轮对话摘要，可升级为持久化记忆层
- **人机回环**：Tester 不通过时让用户介入决策（修 vs 跳过），而非自动循环
- **A/B 模型对比**：相同任务同时跑两个不同模型，对比输出质量
