### 1.2 v3.2 目标

v3.1 完成了核心多智能体协作引擎和 Web 化迁移，但存在两个明显短板：① 前端使用 Bootstrap 5，组件定制能力有限；② 会话数据仅存浏览器 localStorage，无法跨设备、多用户使用。v3.2 聚焦于**生产化**：

| 维度 | v3.1 | v3.2 |
|------|------|------|
| CSS 框架 | Bootstrap 5 CDN | **daisyUI 5 + Tailwind CSS 4** |
| 布局方式 | 固定侧边栏 | **drawer 抽屉式 + 响应式折叠** |
| 数据持久化 | localStorage | **SQLite（WAL 模式）** |
| 用户系统 | 无 | **用户名注册 + 幂等 ID** |
| 会话管理 | 客户端临时 | **服务端持久化 + 历史恢复** |
| 应用启动 | `@app.on_event("startup")` | **lifespan async context manager** |
| API 端点数 | 7 | **18** |

### 2.2 Router 意图分类

`router.py` 是独立于 Agent 编排的轻量分类器，使用 urllib 直接调 DeepSeek API：

```
用户输入 → urllib POST /chat/completions
         → temperature=0, max_tokens=50, thinking disabled
         → 解析 "标签|复杂度"
         → 返回 (task_type, complexity)
```

**5 种标签：** 编程 / 写作 / 分析 / 问答 / 闲聊
**2 种复杂度：** 轻（快车道）/ 重（慢车道）

**关键词覆写（Router 后执行的三层安全网）：**

| 优先级 | 检测 | 覆写 |
|--------|------|------|
| 1 | 搜索/查资料/检索/基于知识库 | complexity=重 |
| 2 | 以"分析"开头 | complexity=重, task_type=分析 |
| 3 | C语言/Java/Rust/Go/C++/C# | task_type=问答, complexity=轻 |

### 2.3 Agent 清单（7 个）

| Agent | 图标 | LLM | 工具 | 职责 |
|-------|:----:|-----|------|------|
| **Planner** | 📋 | DeepSeek V4 Flash | 无 | 拆解任务为步骤清单，输出 task_type |
| **Bot** | 🤖 | DeepSeek V4 Flash | 无 | 快车道直接回复，唯一面客角色 |
| **Retriever** | 🔍 | DeepSeek V4 Flash | `search_knowledge` | 查询 ChromaDB 知识库 |
| **Coder** | 💻 | DeepSeek V4 Flash | `write_file`, `read_file`, `calculate`, `analyze_data`, `visualize_data` | Python 代码 + 数据分析 |
| **Writer** | ✍️ | DeepSeek V4 Flash | `write_file`, `read_file`, `search_knowledge` | 报告/方案/文章 |
| **Tester** | ✅ | DeepSeek V4 Flash | `read_file` | 代码+文档质量评审 |
| **Summarizer** | 📊 | DeepSeek V4 Flash | 无 | 汇总执行过程 → 结构化报告 |

### 2.4 LangGraph 工作流状态

```python
class WorkflowState(TypedDict):
    user_input: str
    lane_mode: str          # "auto" | "fast" | "slow"
    task_type: str          # "编程" | "写作" | "分析" | "问答" | "闲聊"
    complexity: str         # "轻" | "重"
    plan: str
    knowledge: str
    code_or_draft: str
    execution_result: str   # 含 exitcode 的 stdout/stderr
    test_result: str
    fix_count: int          # 当前修复轮数 (max 2)
    agent_messages: Annotated[list, operator.add]
    speaking_log: Annotated[list, operator.add]
    final_output: str
```

**5 个条件路由点：**

| 路由点 | 条件 | 结果 |
|--------|------|------|
| 入口 | `complexity == "轻"` | → Bot |
| 入口 | `complexity == "重"` | → Planner |
| Retriever 后 | `task_type == "写作"` | → Writer |
| Retriever 后 | `task_type in ("编程","分析")` | → Coder |
| Executor 后 | `exitcode == 0` | → Tester |
| Executor 后 | `exitcode ≠ 0` | → Summarizer（或 Coder 修复） |
| Tester 后 | `✅ in test_result` | → Summarizer |
| Tester 后 | `❌ in test_result and fix_count < 2` | → Coder/Writer |
| Tester 后 | `fix_count >= 2` | → Summarizer（强制结束） |

### 2.6 用户与会话管理

v3.2 新增 SQLite 持久化层，使用标准库 `sqlite3`，WAL 模式，外键约束：

```
users 表：  id TEXT PK, name TEXT UNIQUE, created_at TEXT
sessions 表：id TEXT PK, user_id TEXT FK→users, title TEXT,
            messages TEXT (JSON), updated_at TEXT
```

- **用户注册**：幂等设计，同一用户名多次请求返回同一 ID（前端 localStorage 缓存）
- **会话自动保存**：每次对话完成后自动 POST `/api/sessions` 保存全量消息
- **历史恢复**：侧边栏列出用户所有会话，点击即恢复完整对话上下文
- **安全**：所有数据库操作使用 context manager 管理连接，异常自动回滚

## 三、关键设计决策

### 3.1 决策总览

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| 1 | **Streamlit → FastAPI Web** | 全面迁移 | 标准 Web 技术栈，无框架锁定，可扩展 |
| 2 | **Router 恢复 + 关键词覆写** | 独立 LLM 分类器 | 自动化 85% + 安全网兜底 15% |
| 3 | **exitcode 驱动分支** | executor 输出 exitcode → 路由决策 | 确定性信号，不受 LLM 幻觉影响 |
| 4 | **5 任务类型恢复** | 编程/写作/分析/问答/闲聊 | 恢复 v2.x 完整分类，分析类型走专用 Coder |
| 5 | **Bootstrap 5 → daisyUI** | daisyUI 5 + Tailwind CSS 4 | 组件更丰富（drawer/collapse/join）、定制更灵活 |
| 6 | **localStorage → SQLite** | 标准库 sqlite3 + WAL | 跨设备、多用户、数据不丢失、支持并发 |
| 7 | **lifespan 替代 on_event** | async context manager | 避免 FastAPI 弃用警告，资源管理更清晰 |
| 8 | **前端折叠思考** | 默认折叠 + 点击展开 | 减少视觉噪音，用户按需查看细节 |

### 3.2 决策 1：为什么从 Streamlit 迁移到 FastAPI Web

**Streamlit 的限制（v3.0 实际遇到的）：**

1. **页面刷新模型**：每次状态变化需要 `st.rerun()` 全页刷新，无法实现精细 DOM 操作
2. **组件不可定制**：`st.chat_message`、`st.expander` 等组件样式和交互受限
3. **部署耦合**：只能 `streamlit run`，无法融入标准 ASGI 生态（Nginx 反代、Gunicorn worker）
4. **SEO/分享**：纯 SPA 渲染，无法被搜索引擎索引

**FastAPI Web 的改善：**

1. **标准 HTTP**：普通 GET/POST，可用任何 HTTP 工具测试和集成
2. **完全 UI 控制**：HTML/CSS/JS 自由定制，不受框架组件限制
3. **ASGI 生态**：可部署到 Uvicorn、Gunicorn、Daphne 等标准服务器
4. **前后端分离潜力**：当前 Jinja2 渲染，未来可改为 React/Vue 前端 + 同一 API

### 3.3 决策 2：Router 恢复 + 关键词覆写

v3.0 移除了 Router，改为手动选择 — 消除了 15% 误判，但增加了用户认知负担。

v3.1 恢复 Router，但不再单纯依赖它：

```
Router LLM → 关键词覆写（二次修正）→ 最终车道
```

**关键词覆写是安全网 — 实例验证：**

| 场景 | Router | 关键词 | 最终 |
|------|--------|--------|------|
| "查一下知识库里的相关资料" | 问答\|轻 | 搜索检测 → 重 | 慢车道 ✅ |
| "分析这个 CSV" | 分析\|重 | — | 慢车道 ✅ |
| "Java 写一个 Hello World" | 编程\|轻 | 非Python检测 → 问答轻 | 快车道 ✅ |
| "实现 LRU Cache" | 编程\|重 | — | 慢车道 ✅ |

默认自动模式 + 保留手动覆写 = 最佳体验。

### 3.4 决策 3：exitcode 分支 vs 纯文字评审

v3.0 的 executor 执行代码后，将 stdout/stderr 文本传给 Tester 做文字评审。问题在于：Tester 是 LLM，可能误判 — 代码已经跑通了但 Tester 说"看起来有问题"。

v3.1 加入 exitcode 驱动分支：

- `exitcode: 0` → 代码执行成功 → 进入 Tester 评审
- `exitcode ≠ 0` → 代码执行失败 → 返回 Coder 修复（含 stderr）
- 修复超过 2 次 → 强制进入 Summarizer

这是一个确定性信号（exitcode 是操作系统返回的），不受 LLM 幻觉影响。

### 3.5 决策 5：Bootstrap 5 → daisyUI + Tailwind CSS

v3.1 使用 Bootstrap 5 CDN，组件体系相对单薄。v3.2 迁移到 daisyUI 5 + Tailwind CSS 4：

| 维度 | Bootstrap 5 | daisyUI 5 |
|------|------------|-----------|
| 布局组件 | 无 drawer | **drawer 抽屉式侧栏**，移动端友好 |
| 折叠组件 | accordion（JS 依赖） | **collapse（纯 CSS）**，checkbox 驱动 |
| 按钮组 | btn-group | **join**，更紧凑的视觉整合 |
| 弹窗 | modal（JS 依赖） | **modal + dialog**，原生 HTML dialog 兜底 |
| 主题 | 需 Sass 编译 | **30+ 内置主题** + CSS 变量即时切换 |
| 定制 | 覆盖 CSS 类 | Tailwind 原子类 + daisyUI 语义类双层定制 |

迁移后，代码量减少约 30%，组件交互更流畅（drawer 侧栏在移动端自动隐藏）。

### 3.6 决策 6：localStorage → SQLite 持久化

v3.1 的会话数据仅存浏览器 localStorage，存在三个致命问题：

1. **无法跨设备**：换个浏览器或设备，历史对话全部丢失
2. **无法多用户**：同一浏览器只能有一个用户的上下文
3. **数据不可靠**：清理浏览器数据即丢失

v3.2 引入 SQLite 持久化层：

```
users (id, name, created_at)  →  用户身份独立于浏览器
sessions (id, user_id, title, messages, updated_at)  →  会话服务端存储
```

选择标准库 `sqlite3` 而非 ORM（如 SQLAlchemy）的理由：
- 零额外依赖
- WAL 模式支持读写并发
- context manager 保证连接正确关闭和异常回滚

### 5.3 慢车道完整流水线

**输入：** "写一个 Python 函数计算阶乘"
**车道：** 自动 → Router 判定 编程\|重 → 慢车道

| 步骤 | Agent | 产出 | speaking_log |
|:----:|-------|------|-------------|
| — | 🧠 Router | 编程\|重 | — |

**最终产出：** 代码实现 + 执行结果 + 评审结论 + 完整报告。

### 6.1 daisyUI + Tailwind CSS 前端重构

v3.1 使用 Bootstrap 5 CDN，组件体系相对单薄，定制需要覆盖大量 CSS。v3.2 迁移到 daisyUI 5 + Tailwind CSS 4：

| 表单控件 | 标准 input | **input + file-input**，语义化尺寸（xs/sm/md） |
| 主题 | 手动 CSS 变量 | **30+ 内置主题**，一键切换 |
| 加载状态 | 手动 spinner | **loading** 工具类 + **btn 禁用态** |

### 6.2 SQLite 用户与会话持久化

从零构建了完整的数据持久化层（`db.py`，约 160 行），使用标准库 `sqlite3`：

- **WAL 模式**：写不阻塞读，支持页面刷新并发请求
- **外键约束**：sessions.user_id → users.id，保证引用完整性
- **context manager**：自动 commit/rollback/close，异常安全
- **幂等用户注册**：同一用户名多次注册返回同一 ID

API 端点新增 6 个（users CRUD + sessions CRUD），前端自动保存 + 历史恢复。

### 6.3 lifespan 启动管理

v3.1 使用已弃用的 `@app.on_event("startup")`，v3.2 改为标准的 async context manager：

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.db = Database(os.path.join(_PROJECT_DIR, "data.db"))
    yield
```

数据库连接在应用启动时初始化，关闭时自动清理，资源管理更清晰。

### 6.4 安全增强

- **路径遍历防护**：`os.path.basename()` 消毒上传/删除文件名
- **文件类型白名单**：仅允许 pdf/txt/png/jpg/jpeg
- **SQL 参数化查询**：所有数据库操作使用 `?` 占位符，无 SQL 注入风险
- **删除操作 try/except OSError** 保护

### 7.2 关键收获

1. **标准 Web > 框架锁定**：FastAPI + Jinja2 + daisyUI 是 Python Web 的现代组合 — 任何 Python 开发者都能理解和维护，不依赖特定框架的组件模型
2. **自动化 + 安全网 = 最优解**：Router 自动分类覆盖 85% 场景，关键词正则兜底剩余 15% — 纯手动或纯自动都不如两者结合
3. **确定性信号 > LLM 判断**：exitcode 是操作系统返回的确定性值，比 LLM 的"看起来没问题"可靠得多 — 能用代码判断的就不要问 LLM
4. **LangGraph 的显式图优势**：加 exitcode 分支只需加一个 `add_conditional_edges`，清晰可维护
5. **持久化是产品的分水岭**：SQLite + WAL 让系统从"演示 DEMO"升级为"可用的产品" — 用户数据不再丢失，跨设备可访问
6. **daisyUI 的语义化优势**：drawer/collapse/join 等语义组件让 HTML 结构即文档，比 Bootstrap 的通用 div+class 更易维护

### 7.3 可改进方向

- **WebSocket 流式推送**：当前一次性返回，改为 SSE/WS 逐 Agent 推送可大幅提升感知速度
- **Docker 沙箱**：subprocess 在宿主机执行，升级为 Docker 容器隔离更安全
- **并行 Agent 执行**：LangGraph 支持 `Send()` API，可将独立任务并行化
- **前端框架渐进升级**：当前原生 JS 可平滑迁移到 htmx 或 Alpine.js，保持零构建工具
- **A/B 模型对比**：同一任务同时跑两个不同模型，对比输出质量
- **长期记忆**：当前仅注入最近 3 轮摘要，可升级为长期记忆层（如 Mem0 / LangChain Memory）
